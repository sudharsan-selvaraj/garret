Part 1 · Product

Garret — the programmable desktop layer for macOS

Garret turns the empty space behind your windows into glanceable widgets, summons them all above everything with a keystroke, and lets anyone build a widget for it — from a weather card to a full device controller.

·The problem

The info you glance at all day — calendar, PRs, builds, clipboard, a connected phone — lives in a dozen apps, tabs, and menu-bar drips. The context-switch tax is constant, and the most valuable pixels on screen (the desktop) do nothing but hold a wallpaper. And if the widget you need doesn't exist, you're stuck — you can't just build it.

·The three modes

One layer, surfaced at the level of attention you want.

🌤️

Ambient

Widgets live on the desktop layer — above the wallpaper, behind your apps. Always there, never in the way.

calendar · PRs · CI · system load
⚡

HUD

Hit ⌘⇧Space and the whole layer floats above every app — even a full-screen Space — ready to act, then dismiss.

check & act without leaving your app
⌘

Action palettes

Focused, keystroke-summoned tools that do a thing and get out of the way.

clipboard · device control · command runner

·The ecosystem — Garret is a container

Garret ships the layer, the HUD, and the plumbing, then gets out of the way. Anyone builds a widget and shares it as a .garret file. Two tiers, one authoring model, and the tier is chosen by what a widget needs — you don't pick it.

web · sandboxed

Isolated, strict policy, gets exactly the access it declares (a network host, an account, storage) and nothing more. One-click trust.

native · full access

Raw Node in an isolated process — run programs, touch files, talk to USB. Off by default; runs only after a plain "full access, no sandbox" confirmation.

  • one-tap local automations
Part 2 · Building an extension (proposed @garret/sdk)

Authoring guide

One SDK, one project shape, one UI programming model — for both a safe web widget and a full-access native extension. You declare capabilities; Garret derives the tier and the consent.

01The model

An extension is always a UI (a webview you write) and an optional Host (a Node backend, only when you need system access). They talk over one small, typed bridge — request/response, events, streams. The only difference between a web widget and a native extension is whether you ship a host/. Everything the UI can do has the same API either way.

UI (webview, no Node) ⇄ bridge ⇄ Host (your raw-Node backend, full access) ← native UI (webview, no Node) ⇄ bridge ⇄ Platform (Garret-brokered capabilities) ← web
You declare…TierConsent
UI only, or safe caps (storage, service:*, network:<host>)web / sandboxedDisclosure at install
a host/ and a system cap (process/fs/native/network:*)full accessDefault-OFF, typed confirm
Tier rule (resolved — ARCH-06 / Q3): require BOTH The host entry is the runtime signal (spawn a child?); explicit capabilities are the consent signal (what to tell the user). Full-access needs both. A host with no system cap is rejected at install ("host requires a capability"); a system cap with no host is rejected too ("process requires a host"). network:* also requires a host.

05UI API — @garret/sdk/react

Hooks; all auto-clean up on unmount.

useHost<Api, Events>() — call your host

const host = useHost<Api, Events>()
const { items } = await host.list({ dir })          // request/response (typed)
const run = host.run({ argv })                      // stream (typed) → StreamCall

useGarret() — platform capabilities (both tiers)

const g = useGarret()
await g.storage.set('lastDir', dir)          // per-extension (shared across placements)
await g.instanceStorage.set('cursor', 'p3')  // per-placement (isolated) — see §10
const res = await g.fetch('https://api.github.com/…')
g.notify('Done', 'Build finished')
if (!g.inGarret) return <RunInGarretNotice/>   // dev-in-browser fallback

useHostEvent<Events>(channel, handler, deps?)

useHostEvent('changed', ({ dir }) => { if (dir === current) refresh() })

useStream(factory, deps) — streaming without the footgun

const { chunks, result, error, status, cancel } =
  useStream(() => host.run({ argv }), [argv])
// status: 'idle' | 'streaming' | 'done' | 'error' — deps auto-cancel + restart,
// all cleanup handled by the hook (no manual useEffect + .cancel()).

useConfig<T>() — settings (§08)

const [cfg, patchCfg, setCfg] = useConfig<{ repo: string; poll: number }>()
patchCfg({ repo: e.target.value })   // shallow MERGE — other keys untouched
setCfg({ repo: 'nodejs/node', poll: 60 })  // full REPLACE

06Host API — @garret/sdk/host

defineHost hides the entire bridge envelope + sends ready only after your (possibly async) factory resolves. Cross-reference sibling methods with function declarations — they hoist, so no extra machinery:

host/index.ts
import { defineHost } from '@garret/sdk/host'
import { readdir } from 'node:fs/promises'

export default defineHost<Api, Events>((ctx) => {
  // function declarations hoist → safe to cross-reference (the P9 fix — no `methods` arg)
  async function list({ dir }: { dir: string }) { return { items: await readdir(dir) } }
  function run({ argv }: { argv: string[] }) {
    return ctx.stream((out, signal) => {
      const child = ctx.spawn(argv, { signal })   // array only, env-scrubbed, auto-tracked
      child.stdout.on('data', d => out.push(d.toString()))
      child.on('close', code => out.end({ code }))
    })
  }
  async function restart(a: { argv: string[] }) { await stop(); return run(a) } // plain call
  function stop() { /* … */ }
  return { list, run, restart, stop }
})

The ctx toolkit

MemberSignatureNotes
ctx.emit(channel, payload)push a typed event to the UI.
ctx.stream(fn) => Streamfn(out, signal); out.push/end/error; signal aborts on cancel/dispose.
ctx.spawn(argv: string[], opts)array only — execFile semantics, no shell. Scrubs GARRET_* from the child env; auto-killed on dispose.
ctx.spawnShell(cmd: string, opts)explicit shell opt-in — you own the injection risk.
ctx.resolveBinary(name, {hint?}) => Promise<string>per-OS PATH probe; throws BINARY_NOT_FOUND with a platform-correct install hint.
ctx.storage/secrets/fetchsee §07same surface as the UI's g.*; unrestricted in a host.
ctx.onDispose(cb)run on teardown (SIGTERM → onDispose → SIGKILL).
SDK-03 — ctx.spawn takes string[], never a raw string A string passed to a shell is an injection surface even in a trusted extension (a compromised g.fetch response could populate a command field). Split it yourself — you're in control — or use ctx.spawnShell and own the risk.

08aStreaming

Declare it in the contract by returning Stream<Chunk, Result> (Result defaults to void); produce with ctx.stream; consume with a StreamCall (or useStream).

shared/api.ts
import type { Stream } from '@garret/sdk'
export interface Api {
  list(a: { dir: string }): { items: string[] }
  run(a: { argv: string[] }): Stream<string, { code: number }>   // Chunk, Result
}
export interface Events { changed: { dir: string } }
const run = host.run({ argv })   // StreamCall — NOT a promise (can't await an endless stream)
run.onData(c => append(c)).onEnd(({ code }) => done(code)).onError(showErr)
// run.cancel()  ·  await run.result()  (bounded streams only)
Guarantees (from the review + critic passes) Stream start returns synchronously (can't be killed by the request timeout) · pre-.onData chunks are bounded-buffered + a stream nobody consumes auto-cancels · StreamCall is deliberately not thenable · backpressure coalesces; filter firehoses host-side.

08bConfig & settings

Declare a schema; Garret renders the settings form and persists per-instance values. A defineConfig() TS helper gives inference without a runtime dep.

"config": {
  "repo":  { "type": "string",  "label": "Repository", "default": "nodejs/node" },
  "poll":  { "type": "number",  "label": "Refresh (s)", "default": 60, "min": 10 }
}
config vs storage vs secrets config = user-editable settings for this placement (schema-driven form). storage/instanceStorage = the extension's own state. secrets = the same, encrypted.

09Errors

import { GarretError } from '@garret/sdk'
try { await host.devices() }
catch (e) { if (e instanceof GarretError && e.code === 'BINARY_NOT_FOUND') showInstall(e.hint) }

Codes: BINARY_NOT_FOUND (carries hint) · NOT_FOUND · PERMISSION · UNAVAILABLE (cap off / not in Garret / vault down) · BAD_ARGS · TIMEOUT / CANCELLED · NETWORK · INTERNAL.

10Lifecycle & instances

  • One host per placed instance. Removal/disable → SIGTERM → ctx.onDispose → spawned children auto-killed → SIGKILL after 3s.
  • config is per-instance; storage/secrets are per-extension (shared across placements) — atomic + key-level merged. Same-key writes from two placements still clobber → use g.instanceStorage for per-placement state (SDK-06).
  • Payloads are structured-clone (Date/Map/Uint8Array survive; functions/class instances/handles don't). Binary travels as Uint8Array.
  • Persistence lives outside the integrity-checked code dir; removed on uninstall.
  • Power: when the board is ambient (not HUD), polling is throttled and (via the SDK active signal) widgets should pause rAF/animations.

12Build & package

npm run build   # vite → dist/ui · esbuild → dist/host/index.cjs · pack → my-ext.garret
CSP-safe UI build (baked into the scaffold) The native UI origin runs under script-src 'self' 'unsafe-inline'; connect-src 'none'. The scaffold sets build.modulePreload.polyfill = false (the polyfill uses fetch → blocked), forbids WASM in the UI, and emits same-origin hashed assets only. Do network in the host or via g.fetch.
Native addons (.node) — the open gate Compiled addons (node-pty/usb) aren't supported by the package pipeline yet (per-platform ABI). MVP: shell out to external binaries via ctx.spawn. When added, addons ship per-platform (addon.darwin-arm64.node, …) and Garret loads the right one — see Architecture / cross-platform.

13Full example — Device Control

shared/api.ts
import type { Stream } from '@garret/sdk'
export interface Device { serial: string; model: string }
export interface Api {
  devices(): { devices: Device[] }
  mirror(a: { serial: string }): Stream<string, { exit: number }>
}
export interface Events { deviceAdded: Device; deviceRemoved: { serial: string } }
host/index.ts
import { defineHost } from '@garret/sdk/host'
export default defineHost<Api, Events>((ctx) => {
  const track = ctx.spawn(['adb', 'track-devices'])
  track.stdout.on('data', () => { /* diff → ctx.emit('deviceAdded' | 'deviceRemoved', …) */ })
  ctx.onDispose(() => track.kill())
  async function devices() {
    const adb = await ctx.resolveBinary('adb')   // BINARY_NOT_FOUND w/ per-OS hint
    return { devices: parse(await ctx.spawn([adb, 'devices', '-l']).text()) }
  }
  function mirror({ serial }: { serial: string }) {
    return ctx.stream((out, signal) => {
      const p = ctx.spawn(['scrcpy', '-s', serial], { signal })
      p.stderr.on('data', d => out.push(d.toString()))
      p.on('close', code => out.end({ exit: code }))
    })
  }
  return { devices, mirror }
})
ui/App.tsx
const host = useHost<Api, Events>()
const [devices, setDevices] = useState<Device[]>([])
const load = () => host.devices().then(r => setDevices(r.devices))
useEffect(load, []); useHostEvent('deviceAdded', load); useHostEvent('deviceRemoved', load)
return <ul>{devices.map(d =>
  <li key={d.serial}>{d.model}
    <button onClick={() => host.mirror({ serial: d.serial }).onData(console.log)}>▶</button>
  </li>)}</ul>
Part 3 · Core architecture

Architecture & decisions

The foundations the SDK sits on, ordered by cost-to-change. Full detail + file references live in docs/architecture.md; this is the summary + the resolved decisions.

·The six pillars

#PillarStatus
1Window level model✅ solved — native addon (kCGDesktopIconWindowLevel/kCGScreenSaverWindowLevel + CanJoinAllSpaces|Stationary|IgnoresCycle + non-activating panel + visibleOnFullScreen), stronger than pure setAlwaysOnTop
2IPC wire protocol🟡 envelope done (structured clone, per-instance ids); adopt first-class stream_* frames + bridge-level backpressure before the SDK
3Renderer isolation⚠️ on <webview> with per-partition session enforcement in main; WebContentsView is the modern path — the one genuine fork (geometry-sync cost measured via a spike)
4Host process lifecycle🟡 utilityProcess.fork + ready-timeout + stderr + crash-surface done; graceful SIGTERM→onDispose→SIGKILL + GARRET_* env-scrub pending
5Capability enforcement✅ solved — main-enforced (session.onBeforeRequest, permission deny), HMAC-signed consent, re-consent on change
6Power / visibility🟡 HUD-gated poll throttle + backgroundThrottling built; SDK active signal + a true occlusion source pending

·Locked decisions

Q1Package: @garret/sdk (scoped) with /react, /host, /ui — replaces garret-widget-sdk + garret-ext.
Q2Manifest: garret.manifest.json (no collision with other tools).
Q3Tier: require BOTH a host entry and an explicit system capability; reject invalid combos at install.
Q4Frameworks: React (/react) + a 0-dep /ui at launch; Svelte/Vue via community packages if demand appears.
Q5Streams: keep Stream<Chunk, Result>; Result defaults to void.
Q6Config: JSON mini-schema in the manifest + a build-time defineConfig() TS helper (no Zod runtime).
Q7Migration: a compat shim for existing sandboxed widgets at launch; sunset in ~6 months.
Q8Web-tier host: no — "has host ⇒ full access" is absolute for MVP.
Still to lock before writing SDK code (1) Renderer primitive — <webview> vs WebContentsView (the hardest to reverse; geometry-sync cost now measured). (2) Streaming wire — adopt stream_* frames + bridge-level backpressure. (3) Host shutdown — commit the SIGTERM→onDispose→SIGKILL ladder + env-scrub.

·Cross-platform (future) — macOS-only today

Garret is macOS-only now (native mac_window.mm, safeStorage, Homebrew hints). The desktop layer (Pillar 1) is the hard divergence — every OS needs a different trick, and Wayland/GNOME has no clean path yet.

FeaturemacOSWindowsLinux X11Wayland
Ambient desktop layer✅ native🟡 WorkerW🟡 _NET_WM…_DESKTOP❌ compositor-specific
HUD (above all)✅✅✅ _STATE_ABOVE🟡 layer-shell
Overlay on full-screen✅❌🟡🟡
Secrets✅ safeStorage✅ DPAPI🟡 libsecret (daemon)🟡 same
Binary hints✅ brew🟡 scoop/choco✅ apt/dnf✅ same

Design implication: a single GarretPlatform interface (window sink/raise + supportsDesktopLayer/supportsFullscreenOverlay flags, secrets, resolveBinary with a per-OS probe+hint table, autostart, tray) with one impl per OS. resolveBinary's hint (brew install …) is wrong everywhere else — it needs a per-OS table. Order: macOS → Windows → Linux X11 → Wayland (post-1.0).

·Audit adjudication

An external 27-finding audit was reviewed against the code. Nothing dismissed blindly:

  • Already satisfied by the code (auditor saw only the docs): ARCH-03 capabilities-in-main (session enforcement + signed consent); ARCH-05 Mission Control flags (Stationary|IgnoresCycle|CanJoinAllSpaces already set); OS-02 cross-platform secrets (already on safeStorage).
  • Valid — folded into this doc + the SDK spec: ARCH-06 tier-require-both · SDK-01 @garret/sdk · SDK-02 function-declaration sibling calls · SDK-03 array-only spawn + spawnShell · SDK-04 patchCfg/setCfg · SDK-05 useStream · SDK-06 g.instanceStorage · SDK-07 typed g.service<T> · SDK-08 doc fixes (P8 restored, config §-ref, defineManifest/g.window//ui shown) · Q1–Q8.
  • Valid direction, already tracked: ARCH-01 WebContentsView (= Pillar 3 open fork; our enforcement is already session-level, not attribute-CSP); ARCH-02 stream_* wire (= Pillar 2); ARCH-04 transport (we use the Electron-native utilityProcess.fork; env-scrub + shutdown ladder are the tracked gaps).
  • Valid future (macOS-only today): OS-01 desktop-layer per-OS · OS-03 resolveBinary per-OS · OS-04 native-addon per-platform builds · OS-05 GarretPlatform abstraction.
