The programmable desktop layer for macOS

Your desktop, turned into a live surface.

Garret turns the empty space behind your windows into glanceable widgets, summons them all above everything with a keystroke, and lets anyone build a widget for it — from a weather card to a full device controller.

The problem

Your context is scattered. Your desktop is dead space.

The information you glance at all day — calendar, PRs, builds, clipboard, a connected phone — lives in a dozen apps, tabs, and menu-bar drips. Meanwhile the most valuable pixels on your screen, the desktop itself, do nothing but hold a wallpaper.

  • The context-switch tax. Cmd-Tab, hunt for the tab, lose your place — dozens of times an hour, just to check something.
  • Menu-bar clutter that's too small to read and too shallow to act on.
  • Dashboards you have to go to. The info should come to you, ambiently, where your eyes already rest.
  • Closed tools. If the one widget you need doesn't exist, you're stuck — you can't just build it.

Open by design

Garret is a container. The widgets are yours.

Garret ships the layer, the HUD, and the plumbing — then gets out of the way. Anyone can build a widget and share it as a .garret file. Two tiers, one authoring model, and the tier is chosen by what a widget actually needs.

web widget · sandboxed

Safe, one-click, limited

Runs isolated with a strict security policy. Gets exactly the access it asks for — a network host, a connected account, local storage — and nothing more.

  • Weather, stocks, RSS, dashboards
  • Connected-account cards (issues, calendar, mail)
  • Install and trust with a glance
native extension · full access

Full system power, honest consent

Raw Node in an isolated process — run programs, touch the filesystem, talk to USB. Off by default; runs only after a plain-spoken "full access, no sandbox" confirmation.

  • Device control (mirror & drive a phone)
  • Terminals, file explorers, USB tools
  • System monitors, local automations

You don't pick a tier — you declare capabilities, and Garret derives the runtime and the consent.  See the authoring guide →

Principles

What makes Garret different

🪟

Native-feeling

A real desktop layer and a non-activating HUD that floats over full-screen Spaces — not a floating browser window bolted on.

🔒

Honest security

Two tiers with consent that matches the risk. Full-access is off by default and says exactly what it means. Secrets are encrypted; the enable state can't be forged.

🧩

Genuinely open

No gatekept store for system-access code. Build a widget, share a file, install by choosing to trust — like a Homebrew formula, not an app-store submission.

🏠

Local & private

Your data and tokens stay on your machine. Widgets get only the access you grant, and you can see and revoke it.

Under the hood

How it fits together

One board window that lives on the desktop level and rises to a HUD on command; widgets run in tiers matched to their trust.

⌘⇧Space ─────────────▶ raise the board above all apps (HUD) / sink to desktop level (ambient) Garret board (one window) ├─ built-in widgets first-party, in-process ├─ web widgets isolated webview · strict policy · capability broker └─ native extensions UI webview ⇄ bridge ⇄ raw-Node host (full access, own process) secrets → OS keychain · install records signed · per-widget data, sandboxed & GC'd
