Building a Garret Extension

One SDK, one project shape, one UI programming model — for both a safe web widget and a full-access native extension. You don't pick a "type"; you declare capabilities, and Garret derives the runtime tier (sandboxed vs. full-access) and the consent it requires.

01The model

A Garret extension is always the same two things: a UI (a web view you write in React/HTML) and an optional Host (a Node backend you write only when you need system access). They talk over one small, typed bridge — request/response, events, and streams. That's the entire mental model, and it's identical across tiers.

UI (webview, no Node) ⇄ bridge ⇄ Host (your raw-Node backend, full access) ← native extension UI (webview, no Node) ⇄ bridge ⇄ Platform (Garret-brokered capabilities) ← web widget

The only difference between a web widget and a native extension is whether you ship a host/. Everything the UI can do — call methods, subscribe to events, consume streams, use storage/secrets/fetch — has the same API either way. This is the unification: no two SDKs, no branching authoring path. You write one extension; capabilities decide the tier.

How the tier is derived

You declare…TierConsentRuntime
UI only, or safe caps (storage, service:*, network:<host>)web / sandboxedDisclosure only, at installIsolated webview, strict CSP, capability-gated broker
a host/, or a system cap (process, fs, network:*, native)full accessFull-access, default-OFF, typed confirmRaw-Node host process + isolated UI webview
Why derive, not declare An author shouldn't have to know the word "sandbox." They ask for what they need (capabilities); Garret picks the safest runtime that satisfies it and asks the user for exactly the consent that tier warrants. Escalating a capability (e.g. adding process) re-triggers consent.

03Quick start

The same shape, side by side. The native one is the web one plus a host.

web widget  GitHub stars

garret.manifest.json
{
  "id": "gh-stars",
  "name": "GitHub Stars",
  "version": "0.1.0",
  "capabilities": ["network:api.github.com", "storage"],
  "ui": "dist/ui/",
  "defaultSize": { "w": 3, "h": 2 }
}
ui/App.tsx
import { useGarret } from 'garret/react'
import { useEffect, useState } from 'react'

export default function App() {
  const g = useGarret()               // platform caps
  const [stars, setStars] = useState<number>()

  useEffect(() => {
    g.fetch('https://api.github.com/repos/nodejs/node')
      .then(r => r.json())
      .then(d => setStars(d.stargazers_count))
  }, [])

  return <div className="card">★ {stars ?? '…'}</div>
}

No host. Runs sandboxed; network: allowlisted.

Has a host + process → full-access consent.

Both UIs call their backend the same way — useGarret() for platform, useHost() for your own host methods. Next sections define every piece.

05UI API — garret/react

Four hooks. All auto-clean up on unmount.

useHost<Api, Events>() — call your host

Returns a typed proxy of your host's methods. A method typed to return Stream<C> returns a StreamCall; anything else returns a Promise.

const host = useHost<Api, Events>()

const { items } = await host.list({ dir })          // request/response (typed)
const run = host.run({ cmd: 'ping -c3 1.1.1.1' })   // streaming (typed)
run.onData(line => append(line)).onEnd(() => done())

useGarret() — platform capabilities

The Garret-brokered APIs (§06). Available in both tiers; identical surface. In the host, the same object is ctx.

const g = useGarret()
await g.storage.set('lastDir', dir)
const res = await g.fetch('https://api.github.com/…')   // capability-gated
g.notify('Done', 'Build finished')

useHostEvent<Events>(channel, handler, deps?)

Typed subscription to host-pushed events. Payload type comes from Events.

useHostEvent('changed', ({ dir }) => { if (dir === current) refresh() })
useHostEvent('deviceAdded', d => setDevices(xs => [...xs, d]))

useConfig<T>() — the widget's settings (§07)

const [cfg, setCfg] = useConfig<{ repo: string }>()
<input value={cfg.repo} onChange={e => setCfg({ repo: e.target.value })} />
No-bridge fallback (dev in a browser) Outside Garret, useGarret()/useHost() return a stub that throws a friendly GarretError('UNAVAILABLE') on call and reports g.inGarret === false, so you can render a "run inside Garret" state instead of a blank screen. (This is what kills the P10 blank-screen failure mode.)

06aHost API — garret/host

defineHost hides the entire bridge envelope (the {t:'req'} / {t:'res'} / ready plumbing you'd otherwise copy-paste) and gives every method a toolkit ctx. The second arg is the resolved method table so a method can call a sibling safely.

host/index.ts
import { defineHost } from 'garret/host'
import { readdir } from 'node:fs/promises'
import type { Api, Events } from '../shared/api'

export default defineHost<Api, Events>((ctx, methods) => ({
  // request/response — just return (or throw a GarretError)
  list: async ({ dir }) => ({ items: await readdir(dir) }),

  // streaming — return ctx.stream(...)
  run: ({ cmd }) => ctx.stream((out, signal) => {
    const child = ctx.spawn(cmd, { signal })          // env-scrubbed, auto-tracked
    child.stdout.on('data', d => out.push(d.toString()))
    child.on('close', code => out.end({ code }))
  }),

  // sibling calls via the `methods` table (never a bare identifier)
  restart: async (a) => { await methods.stop(); return methods.run(a) },

  stop: () => { /* … */ },
}))

The factory may be async; ready is sent only after it resolves. Throwing during setup surfaces on stderr (piped, prefixed) instead of hanging the UI.

07Streaming

A request/response call returns once; a stream produces values over time (process output, adb logcat, download progress, log tails). Declare it in the contract by returning Stream<Chunk, Result>, produce it with ctx.stream, consume it with a StreamCall.

shared/api.ts
import type { Stream } from 'garret'
export interface Api {
  list(a: { dir: string }): { items: string[] }        // plain
  run(a: { cmd: string }): Stream<string, { code: number }>  // stream: Chunk=string, Result={code}
}
export interface Events { changed: { dir: string } }
ui/App.tsx
const run = host.run({ cmd })          // returns a StreamCall (NOT a promise)
run
  .onData(chunk => append(chunk))      // 0..n
  .onEnd(({ code }) => markDone(code)) // once
  .onError(err => showError(err))      // GarretError
// run.cancel()                        // aborts the host's signal + kills spawned children
// await run.result()                  // for BOUNDED streams only → resolves on end
What the SDK guarantees (from the critic pass)
  • The stream's start returns synchronously — it can't be killed by the request timeout.
  • Chunks that arrive before .onData attaches are buffered with a bound (drop-oldest + an overflow flag); a stream nobody consumes is auto-cancelled.
  • StreamCall is deliberately not thenable — you can't accidentally await an endless stream and hang.
  • Backpressure: out.push coalesces. For true firehoses (logcat), filter host-side (e.g. grep in the spawn) — documented, not pretended.

08Config & settings

Declare a settings schema in the manifest; Garret renders a settings form and persists per-instance values. Read/write with useConfig.

garret.manifest.json
"config": {
  "repo":   { "type": "string",  "label": "Repository", "default": "nodejs/node" },
  "poll":   { "type": "number",  "label": "Refresh (s)", "default": 60, "min": 10 },
  "compact":{ "type": "boolean", "label": "Compact",     "default": false }
}
const [cfg, setCfg] = useConfig<{ repo: string; poll: number; compact: boolean }>()
Config vs. storage vs. secrets config = user-editable settings for this instance (schema-driven form). storage = the extension's own state (cursors, cache), per-extension, not user-facing. secrets = the same, but encrypted (tokens).

09Errors

Everything across the bridge rejects with a typed GarretError — code + message — so the UI can branch (e.g. offer brew install on BINARY_NOT_FOUND).

import { GarretError } from 'garret'
// host: throw new GarretError('BINARY_NOT_FOUND', 'adb not found', { hint: 'brew install android-platform-tools' })
// ui:
try { await host.devices() }
catch (e) {
  if (e instanceof GarretError && e.code === 'BINARY_NOT_FOUND') showInstall(e.hint)
}
CodeMeaning
BINARY_NOT_FOUNDa required external binary isn't on PATH (carries hint).
NOT_FOUNDfile/resource missing.
PERMISSIONOS denied (TCC, file perms).
UNAVAILABLEcapability off / not in Garret / secrets vault unavailable.
BAD_ARGSinvalid method args.
TIMEOUT / CANCELLEDrequest timed out / stream cancelled.
NETWORKfetch failed or blocked by allowlist.
INTERNALhost threw an unclassified error (message preserved).

10Lifecycle & instances

  • One host per placed instance. Two copies of your widget on the board = two isolated host processes. Board removal / disable → host killed → ctx.onDispose fires → ctx.spawn children auto-killed.
  • Config is per-instance (each placement has its own settings). storage/secrets are per-extension (shared across instances) — writes are atomic and key-level merged, so two instances don't clobber each other's unrelated keys.
  • Payloads are structured-clone (Date, Map, Uint8Array survive; functions, class instances, live handles do not). Binary travels as Uint8Array — no base64 bloat.
  • Persistence across restarts: whatever you put in storage/secrets. Data lives outside the integrity-checked code dir, so it never trips tamper detection, and is removed when the extension is uninstalled.

11Gotchas the SDK handles for you

Every item below is a real bug we hit hand-writing extensions. The unified SDK removes each by construction — this is the argument for the SDK over raw bridge code.

#Raw-bridge footgunHow the SDK removes it
P1Streaming faked with hand-rolled ids + manual UI correlation.First-class ctx.stream / StreamCall; the SDK owns ids + teardown.
P2Events = one untyped firehose; hand-filter by channel string.Typed useHostEvent(channel, cb) with payload types + auto-unsub.
P3Stringly-typed methods; typos are runtime errors; no autocomplete.One shared Api type → typed defineHost + useHost.
P4~15 lines of envelope boilerplate copy-pasted per host.defineHost owns it; forgetting ready is impossible.
P5Intervals / watchers / children leaked; no teardown hook.ctx.onDispose + auto-tracked ctx.spawn.
P6No storage → invent a ~/.dotfile; racy, plaintext, orphaned.storage/secrets: namespaced, atomic, encrypted, GC'd.
P7Errors flattened to strings; no "install adb" affordance.Typed GarretError + ctx.resolveBinary hints.
P9Sibling method call → ReferenceError: stop is not defined.defineHost((ctx, methods) => …) — call methods.stop().
P10Inline UI script redeclares a global (const garret, let parent) → whole script silently aborts.Module-scoped bundled UI + import — a global can't be redeclared.
—Host crash invisible; a stuck host hangs the UI forever.Host stderr piped & prefixed; ready-timeout surfaces startup failures.

12Build & package

One command builds both bundles and packs a .garret. The scaffold ships a CSP-safe config so you can't hit the blank-screen traps.

npm run build     # vite → dist/ui  ·  esbuild → dist/host/index.cjs  ·  pack → my-ext.garret
npm run dev       # live-reload against a dev Garret
CSP-safe UI build (baked into the scaffold) The native UI origin runs under script-src 'self' 'unsafe-inline'; connect-src 'none'. The scaffold's vite config sets build.modulePreload.polyfill = false (the polyfill uses fetch → blocked), forbids WASM in the UI (no wasm-unsafe-eval), and emits same-origin hashed assets only. Don't fetch remote scripts/styles from the UI — do network in the host or via g.fetch.
Native addons (.node) — the open gate node-pty/usb-style compiled addons are not yet supported by the package pipeline (ABI pinning). MVP path: shell out to external binaries via ctx.spawn (adb, scrcpy). Terminals/USB via addons is tracked separately.

13Full example — Device Control

A complete full-access extension: list Android devices (hotplug events), stream scrcpy logs, launch a mirror. Shows contract + host + UI together.

shared/api.ts
import type { Stream } from 'garret'
export interface Device { serial: string; model: string; state: string }
export interface Api {
  devices(): { devices: Device[] }
  mirror(a: { serial: string }): Stream<string, { exit: number }>   // scrcpy logs stream
}
export interface Events { deviceAdded: Device; deviceRemoved: { serial: string } }
host/index.ts
import { defineHost } from 'garret/host'
import { GarretError } from 'garret'
import type { Api, Events } from '../shared/api'

export default defineHost<Api, Events>((ctx) => {
  // hotplug → typed events
  const track = ctx.spawn(['adb', 'track-devices'])
  track.stdout.on('data', () => { /* diff + ctx.emit('deviceAdded' | 'deviceRemoved', …) */ })
  ctx.onDispose(() => track.kill())

  return {
    devices: async () => {
      const adb = await ctx.resolveBinary('adb')  // throws BINARY_NOT_FOUND w/ brew hint
      const out = await ctx.spawn([adb, 'devices', '-l']).text()
      return { devices: parse(out) }
    },
    mirror: ({ serial }) => ctx.stream((out, signal) => {
      const p = ctx.spawn(['scrcpy', '-s', serial], { signal })
      p.stderr.on('data', d => out.push(d.toString()))
      p.on('close', code => out.end({ exit: code }))
    }),
  }
})
function parse(_s: string) { return [] }
ui/App.tsx
import { useHost, useHostEvent, useGarret } from 'garret/react'
import { GarretError } from 'garret'
import type { Api, Events } from '../shared/api'
import { useEffect, useState } from 'react'

export default function App() {
  const host = useHost<Api, Events>()
  const [devices, setDevices] = useState<Api['devices'] extends never ? never : any[]>([])
  const [err, setErr] = useState<string>()

  const load = () => host.devices().then(r => setDevices(r.devices))
    .catch((e: GarretError) => setErr(e.code === 'BINARY_NOT_FOUND' ? e.hint! : e.message))
  useEffect(load, [])
  useHostEvent('deviceAdded', load)
  useHostEvent('deviceRemoved', load)

  if (err) return <div className="err">{err}</div>
  return (
    <ul>{devices.map(d => (
      <li key={d.serial}>
        {d.model}
        <button onClick={() => host.mirror({ serial: d.serial }).onData(console.log)}>▶</button>
      </li>
    ))}</ul>
  )
}

14Review checklist (decide before implementation)

Open syntax / design questions for you
  1. Package name & subpaths. garret with /react, /host, /ui? Or @garret/sdk? (Replaces both garret-widget-sdk and the planned garret-ext.)
  2. Manifest filename. garret.manifest.json vs keeping manifest.json (current native/sandbox tiers use manifest.json with kind). Unify how?
  3. Tier inference. Infer full-access from presence of host/, or require an explicit process/fs/native capability too (current spec)?
  4. UI framework. First-class React (garret/react) + a framework-agnostic garret/ui. Enough, or also Svelte/Vue adapters?
  5. Stream chunk typing. Stream<Chunk, Result> — keep Result (the end value), or streams end void?
  6. Config schema. Bespoke mini-schema (shown) vs a subset of JSON Schema vs Zod-in-TS?
  7. Migration. Do we rewrite the existing sandboxed garret-widget-sdk widgets onto this, or keep a compat shim?
  8. Web-tier host. Should a web widget ever have a (sandboxed, capability-limited) host, or is "has host ⇒ full access" absolute?
